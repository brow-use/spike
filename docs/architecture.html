Architecture

How a prompt becomes browser automation

The agent loop, the MCP server, two execution modes, and the layered command system.

The four moving parts

claude-code
Agent

Reasons about the scenario, calls MCP tools, observes results, decides the next step. Every command is a markdown file Claude executes verbatim.

mcp/
MCP server

Long-running Node.js process. Owns the live browser session, exposes ~25 tools, and routes calls to either Playwright or the extension based on mode.

tool/
Tools

Browser tools (navigate, click, get_accessibility_tree…) and file-only tools (write_page_object, write_feature_doc, record_run…). Each tool is a single TypeScript module.

extension/
Chrome extension

MV3 extension. Connects to the MCP server via WebSocket and proxies commands into your real Chrome tab using playwright-crx.

The agent loop

Every brow-use command is a loop, not a script. Claude reads the current page state via the accessibility tree, decides what to do next based on the command's instructions and the user's intent, calls an action tool, then snapshots the result. The MCP server keeps the browser alive across calls, so each step picks up exactly where the last one left off.

Two execution modes

The same agent, the same tools, the same outputs — but two different ways the browser commands actually reach a browser.

Capability Mode 1 — Playwright Mode 2 — Session
Browser Fresh Chromium launched by Playwright Your real Chrome with your profile
Login state None — must authenticate per run Already logged in
Cookies & storage Empty Real session data
Trace fidelity Full Full — same format, same viewer
Visible indicator Separate browser window appears Yellow "DevTools is debugging" banner
Setup Nothing — default mode Build & load the Chrome extension once

In session mode the MCP server runs a WebSocket server on port 3456. The brow-use extension connects to it as a client. Every browser tool call is forwarded over the socket; the extension executes it via playwright-crx (a full Playwright API backed by chrome.debugger) and returns the result. File-writing tools always run on the Node side regardless of mode.

For the architectural diagram of both modes side-by-side, see the diagram page. For details on the extension itself, see Chrome extension. For session mode setup, see Session mode.

The command layers

Commands are organised into layers. Each layer builds on the artifacts produced by the one below it, but every layer is independently runnable — you don't have to do /bu:explore to use /bu:record-page-objects, and you don't have to generate Page Objects to run /bu:do.

Layer 1   Live capture

Drive the browser to capture raw artifacts: traces, aria-tree logs, screenshots. The user describes intent; the agent executes it.

  • /bu:explore — walk the app autonomously, breadth-first, with loop detection.
  • /bu:run — carry out a one-off plain-English intent and record it.
  • /bu:record-page-objects — walk a specific scenario and emit POMs as the run finishes.
  • /bu:record-workflow — record a flow and write it as a reusable function.

Layer 2   Deterministic post-processing

Read trace zips and turn them into queryable artifacts. No browser, no LLM — pure shell and TypeScript so the same trace always produces the same output.

  • make extract SESSION=<id> — extract aria-tree log + per-step screenshots from a trace.
  • The viewer ingester — turns trace + sidecar + reasoning logs into the run-viewer database.

Layer 3   Knowledge generation

Take Layer 1 + Layer 2 artifacts and produce human and machine knowledge: docs the user reads and Page Objects tests can import.

  • /bu:document — feature docs + a page-transitions index, scoped per explore run.
  • /bu:generate-page-objects — typed POMs from the aria log; uses observed-edges to type navigation methods.

Layer 4   Grounded execution

With docs and POMs in place, run business-level intents against the app — grounded in the knowledge layer below, falling back to the raw artifacts when needed.

  • /bu:do — extract data, follow a flow, satisfy an intent. Output in the format you ask for.
  • /bu:run-workflow — run a recorded workflow function against the live app.

The run database

Every command that drives the browser ends with a record_run call. The result is appended to .brow-use/runs.json — a flat JSON array, one entry per run, that captures:

  • Identity — sessionId, command, app, mode, timestamps.
  • Outcome — pages visited, termination reason, records extracted, intent.
  • Artifacts — paths to the trace, aria log, docs folder, result file.

Downstream commands like /bu:document and /bu:generate-page-objects read runs.json as their list of available source runs. The user picks one and the command resolves all paths from the entry.

Observability

Every long-running command additionally writes to output/reasoning/<sessionId>.jsonl via the log_reasoning tool — one line per non-obvious decision (plan, decision, observation, error). This is the audit trail you read after the fact to understand why the agent did what it did, separate from the trace's what.

The viewer/ app stitches all of this together: trace + sidecar + reasoning + aria log → a navigable timeline of the run with screenshots, decisions, and navigation edges side by side.
