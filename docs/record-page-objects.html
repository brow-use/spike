/bu:record-page-objects

How a user prompt becomes browser automation becomes generated code.

Overview

The skill drives a real browser through a user-described scenario and generates typed Page Object Model classes for every page encountered. Claude Code acts as the agent — reasoning about what to do next, calling MCP tools to act on the browser, and finally writing TypeScript files to disk.

The MCP server

The MCP server is the bridge between the agent and the browser. The agent cannot touch the browser directly — every perception and action goes through the server. It owns the live browser session for the duration of a skill run, so the browser stays open and in the same state between tool calls. This is what makes the agent loop possible: each step picks up exactly where the last one left off.

When a command starts, the agent reads .brow-use/apps.json directly to get the currently active app — its URL, name, and description — before it touches the browser.

The agent loop

The core problem is that writing a page object requires knowing what is on the page — but the only way to know what is on a page is to be there. The plugin solves this by giving the agent a perception tool and a set of action tools, then letting it drive the browser step by step through the scenario the user described.

Before each interaction, the plugin calls get_accessibility_tree to snapshot the live ARIA role tree of the current page. This gives the agent the structure, labels, and interactive elements it needs to decide what to do next — and crucially, what selector to record for each element. Accessible selectors (role, label, placeholder) are preferred over CSS because they survive UI changes and read cleanly inside a page object class. 

snapshot current page via get_accessibility_tree
  ↓
identify the next step of the scenario (click, type, navigate)
  ↓
execute the step — browser state changes
  ↓
snapshot again
  ↓
continue until the full scenario is complete

At each step the agent also records which page it is on, what elements were present, and which action caused a navigation to a new page. This running record is what becomes the name map used in Pass 2. The agent is not replaying a fixed script — it is reasoning about the current state of the browser at every step, which means it handles redirects, conditional UI, and intermediate states naturally.

Two-pass strategy

Page objects reference each other — a navigation method on LoginPage must return a typed DashboardPage instance. In a single pass, that class name is unknown until the page is visited, making forward references impossible. The two-pass approach separates discovery from generation so all class names are known before any file is written.

Pass 1 — Discovery

Walk the full scenario without writing any files

Claude drives the browser through the entire scenario from start to finish. For every page reached, get_accessibility_tree is called and the result is mapped into an in-memory name map: 

URL/route → {
  className:   "LoginPage",        // PascalCase + Page suffix
  fileName:    "login-page",       // kebab-case + -page suffix
  elements:    [...locators],      // interactive elements found
  navigatesTo: ["DashboardPage"]   // other class names from the map
}

No files are written. The map stays in Claude's context only.

Pass 2 — Generation

Write all files with the complete name map available

With every class name resolved, Claude generates each page object. For each entry in the map it calls write_page_object — the MCP tool that writes the TypeScript file to output/page/.

If a file already exists, Claude reads it first and merges only missing elements before overwriting. This makes the skill idempotent: running it again over the same scenario extends existing page objects without destroying them.

Output

Files land in output/page/ relative to the project directory where Claude Code is running.

output/
  page/
    login-page.ts
    dashboard-page.ts
    search-results-page.ts
