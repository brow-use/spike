Claude Code plugin

Drive a real browser. Generate the tests.

brow-use is a plugin for Claude Code that turns plain-English scenarios into Playwright Page Objects, reusable workflow functions, and human-readable feature documentation — by walking the application live in a real browser and recording everything it sees.

Get started → How it works

The problem

Writing browser tests is mostly mechanical. You open the app, inspect the DOM, copy selectors, write a Page Object, write a workflow that uses it, write the test that uses that. Each layer is straightforward in isolation but tedious to assemble — and the result is brittle: a refactor renames a CSS class and a hundred selectors break.

LLM coding assistants can write the code, but they can't see the application. They guess selectors from screenshots or stale documentation, and the gap between "looks plausible" and "actually works in the browser" is exactly where time gets lost.

What brow-use does

brow-use closes that gap. It gives Claude Code a set of MCP tools that drive a real browser, read the live accessibility tree at every step, and write artifacts to disk. The agent reasons about what to do next using the same accessibility tree your users (and screen readers) see — so the selectors it picks are stable, semantic, and survive UI changes.

/bu:explore
Map the app autonomously

The agent walks the app breadth-first, captures an aria-tree log per page, and records the whole run as a Playwright trace.
/bu:document
Generate plain-language docs

From a single explore run, produce one feature doc per cluster of pages — with embedded screenshots and a page-transitions index.
/bu:generate-page-objects
Typed Playwright POMs

Turn captured exploration data into Page Object Model classes with accessible selectors and typed navigation between pages.
/bu:do
Execute plain-English intents

"Export every Excavating Machine as CSV." The agent grounds in earlier docs/POMs/workflows, drives the browser, and writes the result.
/bu:record-page-objects
Scenario-driven page objects

Walk a specific scenario in the browser; emit one POM per page reached. Idempotent — re-running merges new elements into existing files.
/bu:record-workflow
Reusable workflow functions

Record a sequence of interactions as a parameterised TypeScript async function — the composition layer between POMs and tests.

Why it's different

01

Grounded in the live accessibility tree

Every action and every recorded selector is derived from the page's ARIA structure, not from speculation. The output uses getByRole, getByLabel, getByPlaceholder — selectors that survive UI refactors.

02

Two execution modes — fresh Chromium or your real Chrome

Default mode launches a clean Chromium session via Playwright. Session mode routes commands through a Chrome extension into your real, logged-in browser — no test login flow required.

03

Forensic by default

Every run gets a Playwright trace, an aria-tree log, a per-step screencast, and a row in .brow-use/runs.json. You can audit what the agent actually did, replay it in npx playwright show-trace, or feed it back into a later command.

04

Idempotent and layered

Commands compose. An /bu:explore run feeds /bu:document and /bu:generate-page-objects. Their outputs feed /bu:do and /bu:record-workflow. Re-running over the same scenario merges results instead of overwriting them.

Get started

brow-use installs as a plugin into Claude Code. From there, every workflow starts the same way:

/bu:apps                    register an app, point it at a URL
/bu:explore                 walk the app autonomously
/bu:document                feature docs from the exploration
/bu:generate-page-objects   typed POMs from the exploration
/bu:do                      run plain-English intents grounded in the above

The user guide walks each step end-to-end. If you want to see how the pieces fit together first, head to the architecture page.
